# Cclover Skills - 开发指南

## 项目简介

个人 AI Agent 技能集合，遵循 TDD 方法论开发，适用于任何智能体（OpenCode、Claude Code 等）。每个技能针对特定场景，帮助 Agent 更好地完成任务。

## 技术栈

- **技能格式**: Markdown (SKILL.md)
- **开发方法**: TDD (Test-Driven Development)
- **编写工具**: writing-skills 技能

## 项目结构

```
cclover-skills/
├── skills/              # 所有技能必须放在此目录
│   ├── skill-name/
│   │   ├── SKILL.md    # 技能主文件（必需）
│   │   └── *.*         # 支持文件（可选）
│   └── ...
├── AGENTS.md           # 本文件
└── README.md           # 项目说明
```

**重要规则**：所有技能必须放在 `skills/` 目录下，不能放在项目根目录。

## 开发规则

### 编写技能

**必须使用 writing-skills 技能编写**，除非用户明确表示不加载技能直接编写。

### 更新 README 文档

**添加或修改任何技能时，所有语言版本的 README 必须同步更新。**

必须更新的内容：
- 在 README.md 中更新技能列表（添加新技能或更新描述）
- 保持所有 README 语言版本中的技能描述一致
- 确保安装说明保持准确

这确保用户无论使用哪种语言都能发现和理解所有可用技能。
### TDD 测试流程

编写技能时必须进行 TDD 测试。**关键步骤**：

1. 使用 writing-skills 编写技能
2. 让子代理测试新技能时：
   - **直接告诉子代理刚编写的技能完整路径**
   - **让子代理全文读取技能文件**
   - 这一步非常重要，确保子代理能正确加载和理解技能

示例：
```
请全文阅读 /path/to/cclover-skills/skills/new-skill/SKILL.md，把它当成技能
```

### 技能设计原则

- **通用性**：技能应适用于所有 AI 助手，不依赖特定环境或工具
- **独立性**：技能可以独立使用
- **可组合性**：技能之间可以有相互依赖关系

### 技能提示词编写指南

技能就是提示词。写好技能 = 写好提示词。编写 SKILL.md 时遵循以下原则：

#### 结构

每个技能应按此顺序组织：
1. **角色/上下文** — 告诉 Agent 它是谁、处于什么场景
2. **任务** — 要做什么，用具体动作描述
3. **约束** — 不能做什么、边界条件、边缘情况
4. **输出格式** — 期望的交付物、结构、风格
5. **示例** — 当行为难以用文字描述时，给出输入→输出的配对

#### 核心原则

- **具体胜过抽象**："检查函数是否有错误处理，建议添加 try-catch 和自定义错误类型" 远好于 "改进代码"
- **正面胜过否定**："只讨论功能特性" 比 "不要提到价格" 更有效 — 否定反而会让 Agent 注意到被禁止的概念
- **示例胜过描述**：一个好例子 > 三段解释。Agent 擅长模式匹配，善用这一点
- **原子化指令**：每条要求独占一行。复合句（"做 X 同时考虑 Y 并兼顾 Z"）容易被部分执行
- **明确边界**：显式声明范围内和范围外的内容。无边界的指令会导致范围蔓延
- **按优先级排序**：最关键的指令放最前面。注意力随长度递减

#### 常见陷阱

- **指令冲突**："简洁回答" + "详细解释每一步" — Agent 会在两者间摇摆。选一个，或明确各自适用场景
- **隐式假设**：如果技能依赖特定文件结构、工具可用性或命名约定，必须显式声明
- **过度堆砌**：塞太多要求会稀释每条指令的执行质量。技能超过约 200 行时，拆分或果断取舍
- **忽略边缘情况**：只测试正常路径。TDD 时用对抗性输入压力测试
- **模糊的成功标准**："让它更好" 无法验证。"所有函数都有包含 @param 和 @returns 的 JSDoc" 才可验证
